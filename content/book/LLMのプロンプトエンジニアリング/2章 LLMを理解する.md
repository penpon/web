# 2.1 LLMとはなにか

LLM: 文字列を入力すると文字列を返すサービス

トレーニングされていないLLMは、初めて動作するとき補完としてランダムなUnicode記号の寄せ集めを生成し、プロンプトと明確な関連性を持たない。トレーニングを得たLLMは、単に文字列を返すのではなく、自然な言語を使ってレスポンスを返す。

トレーニングには計算リソースと時間が必要なので、大体はトレーニング済みの汎用モデルを使用する。場合によってはfine tuningも加える

LLMは大規模なドキュメントの集合を使ってトレーニングされる。
トレーニングデータセットの種類は目的に依存する。
トレーニングデータセットに似た出力を生成する方法を学習する。
トレーニングデータセットに含まれているドキュメントの冒頭部分をプロンプトとして与えると、モデルはそのドキュメントが自然に続くと思われるテキストを生成する。

単にトレーニングデータセットを模倣するのはLLMの目標ではない。LLMが目指しているのはトレーニングデータから見出したパターンを抽象化しトレーニングデータとは異なる新たなプロンプトに対しても柔軟に補完できるようになることです。

単なる丸暗記を防ぐためLLMの内部構造は具体例から抽象概念を導き出しトレーニング手順も多様で繰り返しの少ないデータを用いて、未知のデータで成果を測る仕組みを取り入れている

あるプロンプトがどんな補完を生み出すか知りたい場合、「合理的な人間ならどう応答するか」ではなくて「このプロンプトで始まるドキュメントはどのような流れで続く」を考えるべき
## 2.1.1 ドキュメント補完

トレーニングデータをよく知るほど、そのデータでトレーニングされたLLMの出力について直感を形成しやすくなる
## 2.1.2 人間の思考とLLMの処理の違い

人間の言葉は現実を反映するが、モデルの言葉は人間を反映する

## 2.1.3 ハルシネーション

ハルシネーション: 事実と異なるがもっともらしく見える情報を自信を持って生成してしまう現象
モデルの視点では他の補完と区別がつかないため「勝手に作り話をしないで」というプロンプト指示はほぼ効果がない。その代わりモデルに検証可能な情報を提供させるのが一般的。

出した結論の推論の説明をさせたり
解散可能なデータや参照リンク、検索可能なキーワードや詳細情報を含めるといった方法があります。

# 2.2 LLMが見る世界

モデルにテキストを送信するとまずは「トークン」と呼ばれる複数文字のチャンクに分解されます。トークンは3-4文字長ですがよく使われる単語や文字列にはより長いトークンも存在します。

テキストを読む時、モデルはまず「トークナイザー」に通してトークン列に変換する。そのトークン列がLLMに送られます。LLMはトークン列を生成し、それが再びテキストに変換されてユーザに返される。

## 2.2.1 違いその1: LLMは決定論的なトークナイザーを使用する

人間
文字から単語への変換を曖昧に行っている。目にした文字列と似ている語を推測して認識する。
LLM
決定論的なトークナイザーを使う。
ghostはOpenAIのGPTのトークナイザーでは1つのトークンだけど「gohst」は「g」「oh」「st」の3つに分解される。別物として扱われるため誤字を簡単に見分けられる。

## 2.2.2 違いその2: LLMは文字を1つずつゆっくり確認できない

人間
必要に応じてペースを落として1文字1文字慎重に確認できる
LLM
ゆっくり吟味できない。トークンを分解したり再組立する文法的処理が苦手。文字の並び替えなど。できればトークンを分解して扱うような作業はLLMにさせないほうがよい

## 2.2.3 違いその3: LLMは文字を人間と異なる見方をする

人間
トークンや文字に関して直感的な理解を持っている
LLM
英語の大文字変換に失敗する (OpenAIのtext-babbage-001モデル)

人間にとって大文字Aは小文字aの変形版。しかしLLMが使うトークンでは大文字を含むトークンと小文字を含むトークンは全く異なるもの。多くのトークナイザーはモデルがこれらの対応関係を簡単に学習できるように作られていない。
GPTのトークナイザーは「strange new worlds」を「str」「ange」「 new」「 worlds」の4トークンに分解する。「STRANGE NEW WORLDS」では「STR」「ANGE」「 NEW」「 WOR」「L」「DS」の6トークンになる。「gone」は1トークンですが、「G」「ONE」は2トークンとなります。

## 2.2.4 トークン数を数える

コンテキストウインドウ: LLMが一度に扱えるテキスト量の上限
文字数をトークン数に変換する一般的な公式はありません。テキストやトークナイザーによって異なる。一般的なGPTトークナイザーは英語の自然言語テキストをトークン化すると平均1トークンあたり約4文字程度です。ほとんどのトークナイザーは英語に最適化されている。他言語では効率が下がり、1トークンあたりの文字数が減る傾向にある。

# 2.3 1つのトークンずつ

## 2.3.1 自己回帰モデル (Auto Regressive)

LLMに1回処理させると、統計的に最も可能性の高い「次のトークン」が得られる。その後、このトークンはプロンプトの末尾に付け足されLLMは「新しいプロンプト」を前提として再び統計的に最もありうる次のトークンを求めます。

## temperatureと確率

LLMは「最も可能性の高いトークン」を求める
実際には「ありうる全てのトークンの確率」を計算してからその中から1つを選んでいる
トークンを選択する仕組みは「サンプリング」と呼ばれている

「logprob」（トークン確率の自然対数）という形で返す。logprobが高いほどモデルがそのトークンをより有力と考えていることを意味する。logprobが0を超えることはない。0は「このトークンで確定！」に等しいから。最も有力なトークンのlogprobは-2から0の間

0より大きい「temperature」を用いる。
0以上の数値でモデルにどれくらい「創造的」になってほしいかを示す。
前提: トレーニングデータで51%が「 Buck」、31%が「 Three」で続く場合
0: 最も有力なトークンだけ欲しい時に選ぶ。正確性が最優先のときに選ぶ。
-> 「 Buck」を選択
1: トレーニングデータ上の出現確率分布をそのまま反映させたい場合。
-> 51%の確率で「 Buck」が選ばれて、31%で「 Three」が選ばれる。

temperatureが1を超えるとLLMは酔っ払った出力をすることがあります。
